Why its called a Pull Request

Before I became familiar with GitHub, I wondered why contributing your changes to a repo was called a Pull Request. To me, it seemed like it should be called a Push Request because you want your changes to be reviewed for inclusion into the repo. However, now that I've spent time learning Git and its interaction with GitHub I've learned that the term push is used to propogate changes and the term pull is used to integrate changes. With this understanding, I now get why its called a Pull Request, though it still seems a bit odd to me - it just doesn't bother me anymore :)

Avoiding Chaos With Git Pull and Git Rebase

A common issue that crops up when a team starts working together is the tendency to create version chaos because multiple people are editing the same file at the same time or using `git add` too loosely. Of course, an initial fix is to break up the work so more than one person isn't working on the same file. Another simple fix is to be more disciplined about `git add` by targeting it to the specific file being changed rather than the entire directory - e.g. `git add [filename]`. After doing this, version chaos should reduce, but it is still possible to get out of sync. This is where `git pull` comes in. Ideally, its done before you start work on your branch and before you do a `git add` or `git commit`. This way, if you see updates, you can address them before doing a `git push`. Sometimes, even a well timed `git pull` isn't enough. This is where `git rebase` becomes essential. To learn a lot more, please visit: https://github.com/Johanna206/assignment4/wiki

Two Ways to Undo: Git Reset and Git Revert

Recently, we learned about `git reset` and `git revert` - two very different ways to undo changes you made in error or haste or without consulting all the relevant stakeholders. The way I've internalized the difference between the two commands is that `git reset` really undoes everything after a particular `git commit` while `git revert` undoes a `git commit` by creating a new commit that undoes a specific `git commit` . If we only consider the latest commit, the two have a similar effect on file contents, but different effects on commit history. Based on what I've learned, I would think very hard before using a `git reset` on anything that isn't the latest commit. I'm still learning, so I might change my mind later :)
